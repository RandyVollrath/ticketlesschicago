## Static-analysis findings

### Finding 1 — `heading_stale` signature labels the wrong condition
**Files:** `lib/parking-quality-diagnose.ts:141-144`,
`pages/api/mobile/check-parking.ts:752-765`

**Server-side stale guard** (`check-parking.ts:754-765`) fires when **both**:
- `headingDisagreementDeg > 30°`
- GPS and compass classify to *different* grid orientations
  (`orient(gps) !== orient(compass)`, where `orient()` maps a bearing to
  `'N-S'` or `'E-W'`)

When it fires, `effectiveHeadingSource = 'compass'` and the diagnostic row is
saved with `heading_source: 'compass'`.

**Classifier** (`parking-quality-diagnose.ts:141-144`):
```typescript
if (row.heading_source === 'gps'
    && typeof headingDisagreementDeg === 'number'
    && headingDisagreementDeg > 30) {
  return 'heading_stale';
}
```

This fires when `heading_source === 'gps'`, which means the stale guard **did
NOT** fire — the server correctly kept GPS because both headings mapped to the
same grid orientation. These rows are handled correctly; the disagreement was
above 30° but inconsequential because both pointed roughly the same way.

The rows where the guard actually fired (GPS was stale and compass took over)
have `heading_source === 'compass'` and are **not** labeled `heading_stale` —
they fall through to later checks and may be marked healthy.

**Net effect:** the `heading_stale` diagnostic count is inflated with
correctly-handled rows, and the actual stale-guard-fired rows are invisible
in that signature. Operational investigations triggered by a high
`heading_stale` count would look at the wrong events.

**Proposed patch shape** (needs data validation before implementing):
Change the classifier to match only cases where the stale guard provably fired.
One approach: server writes a `native_meta.heading_stale_guard_fired = true`
flag when it switches to compass at line 762, then the classifier checks
`row.native_meta?.heading_stale_guard_fired === true` instead of inferring
from `heading_source === 'gps'`.

Alternatively, replicate the orientation check in the classifier:
```typescript
function orient(h: number): 'N-S' | 'E-W' {
  const n = ((h % 360) + 360) % 360;
  return (n <= 45 || n >= 315 || (n >= 135 && n <= 225)) ? 'N-S' : 'E-W';
}
// In classify():
const compassH = row.native_meta?.compassHeading;
const gpsH = row.native_meta?.gpsHeading;
if (row.heading_source === 'gps'
    && typeof headingDisagreementDeg === 'number'
    && headingDisagreementDeg > 30
    && typeof compassH === 'number'
    && typeof gpsH === 'number'
    && orient(gpsH) !== orient(compassH)) {
  return 'heading_stale';
}
```
But this requires confirming that `compassHeading` and `gpsHeading` are stored
in `native_meta` — only visible via live DB query. **Do not implement until
confirmed with example rows.**

**Risk:** if the flag or field doesn't exist in older rows, the classifier would
silently under-count `heading_stale` during backfill windows.

### Finding 2 — `snap_far` fires at 20m; server accepts up to 60m
**Files:** `lib/parking-quality-diagnose.ts:132-136`,
`pages/api/mobile/check-parking.ts:1158,1176`

The classifier labels any snap > 20m without Mapbox confirmation as `snap_far`.
The server accepts snaps up to 40m (default) or 60m (when there's a clear
geometric winner or Apple-map confirmation). A snap at 25m with no Mapbox
response is valid by server logic but classified as a failure by the diagnostic.

This may or may not be a problem — the 20m threshold in the classifier is
deliberately conservative. Without knowing the actual street-correct rate for
25-40m snaps, it's impossible to say whether the threshold should move.

**No patch proposed** — pure data question.

### Finding 3 — `streetBearing >= 0` guard in chicago-grid-estimator.ts is always true for valid bearings
**File:** `lib/chicago-grid-estimator.ts` (around line 271)

The guard `snapGeometry.streetBearing >= 0` passes for all valid 0–360° values.
The intent appears to be "bearing is present and valid." For `NaN`, `NaN >= 0`
is false (correct). For `undefined`, `undefined >= 0` is false (correct). The
guard accidentally works for the right reasons, but reads as dead code for the
normal case.

**No functional bug.** Worth cleaning up to `streetBearing != null &&
!isNaN(streetBearing)` for readability if a refactor touches this file.
