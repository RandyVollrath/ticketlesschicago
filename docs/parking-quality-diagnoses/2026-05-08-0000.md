## Headline regression

Nominatim (OSM reverse geocoder) is called with the **raw, uncorrected GPS fix**
(`latitude, longitude`) while the snap pipeline and Mapbox both evaluate the **corrected
position** (`correctedLat, correctedLng`). When a GPS correction offset or CarPlay anchor
shifts the parking point by 10–50 m, Nominatim evaluates a different location than the snap,
producing spurious orientation disagreements that trigger `nominatim_overrode` — even when
the snap's corrected position was correct.

## Root-cause hypotheses

### Hypothesis 1 — Nominatim uses raw GPS while snap/Mapbox use corrected GPS

**Evidence → hypothesis:**

```
pages/api/mobile/check-parking.ts

~line 868-870  (GPS correction applied to correctedLat/correctedLng):
  correctedLat = latitude + corr.offset_lat;
  correctedLng = longitude + corr.offset_lng;

~line 890-894  (CarPlay anchor applied to correctedLat/correctedLng):
  correctedLat = cpDisconnectLat;
  correctedLng = cpDisconnectLng;

line 1108  (snap uses corrected coords — CORRECT):
  user_lat: correctedLat,
  user_lng: correctedLng,

line 1729  (Nominatim uses RAW coords — BUG):
  const nominatimResult = await reverseGeocode(latitude, longitude);

line 2008  (Mapbox uses corrected coords — CORRECT):
  const mbRev = await mapboxReverseGeocode(correctedLat, correctedLng);
```

When CarPlay anchor shifts the GPS point (e.g., the raw fix is 30 m into a cross-street
due to post-park walk drift, and the CarPlay disconnect coords correctly identify the actual
parking location), Nominatim still evaluates the walked-away position, identifies the
cross-street, disagrees with the correct snap, and the system overrides the snap.

The comment on line 2003–2008 explicitly documents the intent for Mapbox: *"By this point
correctedLat/Lng already incorporates any accepted per-block correction and CarPlay anchor
sharpening, so Mapbox should evaluate the same parked point."* Nominatim has no such comment
and is missing the equivalent fix.

### Hypothesis 2 — `snap_far` threshold in diagnosis is lower than the server acceptance threshold

The server accepts snap candidates up to `Math.max(accuracyMeters, 30)` m (floor 40 m when
accuracy is unknown). The diagnosis classifier marks `snap_far` for any snap >20 m when
Mapbox does not confirm. Rows at 21–39 m that are valid server-accepted snaps are
miscounted as failures. This is a quality-metrics false positive, not a detection failure —
but it inflates the `snap_far` signal and may hide real failures beneath noise.

## Proposed patches

### Patch 1 — Pass corrected coordinates to Nominatim (HIGH confidence)

- **Signature addressed:** `nominatim_overrode`
- **Expected impact:** Fewer spurious `nominatim_overrode` events for CarPlay-detected parks
  and GPS-corrected parks; snap+Nominatim agreement rate should increase, particularly for
  users who regularly park where CarPlay anchoring fires.
- **Evidence base:** Code inspection — `check-parking.ts:1729` vs `:2008`. No live rows
  available this run.
- **File + line:** `pages/api/mobile/check-parking.ts:1729`
- **Change shape:** Replace `reverseGeocode(latitude, longitude)` with
  `reverseGeocode(correctedLat, correctedLng)`. One-character diff. Nominatim now evaluates
  the same point as the snap and Mapbox — the best current estimate of where the car parked.
- **Test plan:** After deploy, `nominatim_overrode` rate for CarPlay-detected parks should
  drop. Monitor `pct_nominatim_overrode` in next 3 daily quality reports. Secondary: check
  whether `nominatim_agreed` rate increases. No expected regressions for non-CarPlay,
  non-GPS-corrected parks (for those, `correctedLat/correctedLng === latitude/longitude`).
- **Risk:** Extremely low. `correctedLat/correctedLng` has been validated (50 m correction
  cap, 100 m CarPlay anchor drift cap) before it reaches line 1729. In the no-correction /
  no-CarPlay path, the coordinates are identical to raw GPS so behavior is unchanged.

### Patch 2 — Raise `snap_far` diagnosis threshold from 20 m to 35 m (LOW confidence — no live rows)

- **Signature addressed:** `snap_far` (diagnostic false-positive reduction)
- **Expected impact:** Fewer phantom `snap_far` rows; diagnosis signal better reflects
  real street-identification failures.
- **Evidence base:** Code inspection only — `parking-quality-diagnose.ts:131` vs
  `check-parking.ts:1121`. Needs live data to confirm impact.
- **File + line:** `lib/parking-quality-diagnose.ts:131`
- **Change shape:** Change threshold `> 20` to `> 35` in the `snap_far` classifier guard.
- **Test plan:** Run diagnose script before and after; `snap_far` count should drop. Validate
  against any known ground-truth rows where snap was wrong at 20–35 m.
- **Risk:** Could under-count real far-snap failures if some 20–35 m snaps are genuinely
  wrong. Needs ≥3 labeled rows at 20–35 m to validate direction before implementing.
  **Do not implement Patch 2 until live row data is available.**
