## Code observations (static read only — NOT actionable without live data)

These are structural observations from reading the code, **not proposals**.

### Observation A — `snap_far` condition does not check Nominatim agreement

In `lib/parking-quality-diagnose.ts:131`, the `snap_far` classifier only checks
`mapbox_reverse.agrees_with_snap`. If Nominatim independently agrees with the
snap on a row where Mapbox reverse data is absent, the row is still flagged
`snap_far`. This could inflate the `snap_far` count for areas where Mapbox
reverse coverage is thin. Worth checking in live data: do `snap_far` rows tend
to have `nominatim_street == snap_street`?

### Observation B — classifier precedence puts `no_snap` before `snap_far`

`no_snap` is returned when `!row.snap_street_name` (line 122). If snap produces
a result but the distance is huge, it falls through to the `snap_far` check at
line 131. This ordering looks correct but means any row with a blank
`snap_street_name` will never reach `snap_far` — something to confirm with
live data before assuming the counts are independent.

### Observation C — `heading_stale` check in classify() vs. check-parking.ts

`heading_stale` is classified from `row.heading_source === 'stale'` (line 136).
The question is whether `check-parking.ts` is reliably writing `heading_source`
into the diagnostic row for every check. Worth a spot-check on live rows: are
there rows where `heading_source` is `null` and the heading was actually stale
vs. genuinely missing?
