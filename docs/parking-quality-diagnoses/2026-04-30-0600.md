## Code-inspection finding: `heading_stale` classifier is partially dead code

During the blocked-data run, a static review of the classifier in
`lib/parking-quality-diagnose.ts` revealed a mismatch between what
`check-parking.ts` writes and what the classifier reads.

### What the code actually stores

`pages/api/mobile/check-parking.ts:717` declares:

```typescript
let effectiveHeadingSource: 'compass' | 'gps' | 'none';
```

All assignments confirm only three values ever reach the DB:

| Line | Value written |
|------|---------------|
| 730  | `'compass'` — GPS stale, switched to compass |
| 735  | `'gps'`     — GPS trusted |
| 745  | `'gps'`     — trajectory median used as GPS proxy |
| 749  | `'compass'` — GPS absent, compass fallback |
| 753  | `'none'`    — no heading of any kind |

This is stored verbatim to `parking_diagnostics.heading_source` at line 2774.

### What the classifier checks

`lib/parking-quality-diagnose.ts:109`:

```typescript
if (row.heading_source === 'stale' || row.heading_source === 'none') return 'heading_stale';
```

`'stale'` is never stored → that branch is dead code. The only branch that
actually fires is `row.heading_source === 'none'`, which captures rows where
**no heading was available at all**, not where a heading was stale.

### Why this matters

The signature `heading_stale` is documented as "GPS heading used after a turn"
(the classic stale-heading failure mode: car drives east, turns north, parks —
GPS heading is still pointing east). But:

- Rows where the code detected this and **switched to compass** get stored as
  `heading_source = 'compass'` → they pass through the heading_stale check
  unseen and are classified by whatever downstream signature fires (or healthy).
- Rows where no heading was available at all get stored as
  `heading_source = 'none'` → they are **mislabeled as heading_stale** when
  they are really compass_missing.
- The `compass_missing` check at line 110 (`!row.compass_heading && !row.gps_heading`)
  is rendered unreachable for any row that has `heading_source = 'none'`, since
  heading_stale fires first.

### Impact hypothesis

The diagnosis system is blind to two real scenarios:

1. **Stale-GPS-heading rows where no compass was available**: these get
   classified as `healthy` or another downstream signature rather than
   `heading_stale`. If stale GPS heading is causing wrong-street calls when
   compass is absent (single-heading, stale-from-a-turn scenario), the
   diagnosis will miss it.
2. **No-heading rows are mislabeled**: the `heading_stale` count in reports
   is inflated by pure "no heading" cases that are distinct from the
   stale-heading problem.

### Proposed fix (pending ≥3 row evidence)

The fix has two parts:

**Part A — remove dead `'stale'` check and rename `'none'` branch:**
`lib/parking-quality-diagnose.ts:109`:
```diff
-  if (row.heading_source === 'stale' || row.heading_source === 'none') return 'heading_stale';
+  if (row.heading_source === 'none') return 'compass_missing';
```
This merges the no-heading case into `compass_missing` where it belongs and
removes the dead `'stale'` branch.

**Part B — add detection for actual stale-heading scenario:**
A stale GPS heading that got overridden by compass is distinguishable in the
data: `heading_source = 'compass'` AND `native_meta.headingDisagreementDeg > 30`
AND `native_meta.headingPreferredSource = 'compass'`. If these rows turn out to
be correlated with `user_said_street_wrong`, a new `heading_overridden` signature
(or annotating nominatim_overrode with an `override_reason` field) would let us
track whether the compass-preference decision is helping or hurting.

**Evidence needed before implementing:**
- ≥3 example rows with `heading_source = 'none'` showing they are true
  compass_missing cases (no GPS heading, no compass heading), confirming the
  relabeling is safe.
- OR: ≥1 `user_said_street_wrong` row where root cause is stale GPS heading
  without compass available (would show `heading_source = 'gps'` + heading that
  classifies to wrong orientation), confirming Part B is worth pursuing.
