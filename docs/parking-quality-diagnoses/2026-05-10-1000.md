## Headline finding

The walk-away guard (`pages/api/mobile/check-parking.ts:2892`) false-fires on `stop_start` and
`pre-captured` GPS sources. These are car-stop-time GPS fixes — the highest-quality location signal
in the pipeline. Any heading disagreement when GPS source is `stop_start` is caused by GPS *course*
staleness (the car just turned and the course heading hasn't updated), not by the user walking away
from the car. Suppressing parking notifications for these cases is incorrect.

## Top failure signature

### `walkaway_guard_false_positive` (structural, no live rows)

The walk-away guard fires when compass vs GPS heading disagreement exceeds 50°. The guard correctly
excludes `last_driving` GPS source (car was moving → not post-walk). But it does **not** exclude
`stop_start` or `pre-captured` GPS, even though both are captured at the exact moment the car
stops — making them equally immune to walk-away contamination.

**Canonical case — Belden/Kenmore (diag #173, 2026-05-08):**
- Car parked on W Belden Ave (E-W)
- GPS course: 5° N-S (stale from driving north on Kenmore, just before the left turn onto Belden)
- Compass: 98° E-W (fresh magnetometer reading at park time)
- `headingDisagreementDeg = |5 − 98| = 93° > 50°` → walk-away guard fires
- `nativeLocationSource` for a car that just completed a turn and stopped is typically `stop_start`
- But the GPS *fix* (location) is correctly at the Belden parking spot — not a walk-away location

The walk-away guard was designed to catch cases where the GPS *location* drifts to wherever the
user walked (Adams/Laflin, Southport red-light, Lawrence/Wolcott per commit `96e2d28`). In all
three original false-positive cases, the location buffer was filled with walking samples at
`0.3–2.8 m/s` — producing `current_fallback` GPS, not `stop_start`.

Commit `7fcf483` (2026-05-08) added compass-contradiction logic to fix the Belden street-detection
error. But the heading disagreement signal that enables the compass fix (GPS course stale, compass
fresh) is the same signal that triggers the walk-away false-positive. Both commits interact here.

## Root-cause hypothesis

**File:** `pages/api/mobile/check-parking.ts:2892–2895`

```typescript
// Current code (lines 2892-2895):
const walkAwayLikely =
  headingDisagreementDeg != null
  && headingDisagreementDeg > HEADING_WALKAWAY_THRESHOLD_DEG
  && nativeLocationSource !== 'last_driving';  // last_driving fix isn't post-walk
```

The `last_driving` exclusion reasoning: "this fix is from while the car was moving, so it's not
a post-walk GPS capture." The exact same reasoning applies to `stop_start` (GPS captured at the
car's stop moment) and `pre-captured` (iOS native equivalent, labeled in the source comment at
line 801 as "= stop_start from native side").

**Code comment at line 796–801** (in `diag` accumulator) explicitly documents the quality hierarchy:
```
//   'stop_start'       = locationAtStopStart (GOOD — car-stop-time GPS)
//   'last_driving'     = lastDrivingLocation (OK — last GPS while moving)
//   'current_refined'  = refined to current fresh GPS after proximity check (mixed)
//   'current_fallback' = current GPS at finalize time (BAD — possibly walk-away)
```

`stop_start` is rated **GOOD** (above `last_driving` which is already excluded). The asymmetry
in the walk-away guard — excluding `last_driving` but not `stop_start` — is inconsistent with
this quality hierarchy.

## Proposed patch

### Patch 1 — Exclude stop_start/pre-captured GPS from walk-away guard

- **Signature addressed:** `walkaway_guard_fired` (false positives)
- **Expected impact:** Parking notifications restored for stale-GPS-heading parks with stop_start
  fix; `walkaway_guard_fired` count drops for this class of events
- **Evidence base:** Code comment hierarchy (`stop_start = GOOD`, line 796), exclusion reasoning
  in the existing `last_driving` guard (line 2895), Belden/Kenmore canonical case (diag #173),
  walk-away dataset description in commit `96e2d28` (walking samples at 0.3–2.8 m/s → implies
  `current_fallback`, not `stop_start`)
- **File + line:** `pages/api/mobile/check-parking.ts:2895`
- **Change shape:** Add two more exclusions to `walkAwayLikely`. The current guard already says
  "last_driving isn't post-walk"; extending it to say "stop_start and pre-captured also aren't
  post-walk" is the same reasoning applied consistently.

```diff
 const walkAwayLikely =
   headingDisagreementDeg != null
   && headingDisagreementDeg > HEADING_WALKAWAY_THRESHOLD_DEG
-  && nativeLocationSource !== 'last_driving';  // last_driving fix isn't post-walk
+  && nativeLocationSource !== 'last_driving'     // last_driving fix isn't post-walk
+  && nativeLocationSource !== 'stop_start'       // stop_start is car-stop-time GPS — heading disagreement is course staleness, not walk-away
+  && nativeLocationSource !== 'pre-captured';    // iOS native pre-captured = stop_start equivalent
```

- **Test plan:**
  1. Replay diag #173 (Belden/Kenmore) via `scripts/parking-replay-suite.ts` after deploy — the
     replayed event should have `walkaway_guard_fired: false` (or absent) if location_source is
     `stop_start`/`pre-captured`
  2. Confirm the next daily quality report shows reduced `walkaway_guard_fired` count without a
     corresponding increase in user-reported wrong-street cases
  3. The three original walk-away false positives (Adams/Laflin, Southport, Lawrence/Wolcott)
     all had buffer speed < 3 m/s (walking) → `current_fallback` source → still caught by guard
- **Risk:** Very low. Only affects events where GPS source is `stop_start`/`pre-captured`. No
  detection logic changes — the resolved address is unaffected. Only the suppression decision
  changes. If any stop_start fix is genuinely at a walk-away location (theoretically possible if
  CoreMotion transitions out of automotive prematurely), the user would receive a notification for
  a wrong location and need to verify in-app — same UX as if the guard never existed.
