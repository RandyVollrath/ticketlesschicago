## Headline regression

`snap_far` masks `nominatim_overrode` whenever Nominatim overrides the snap to a street whose centerline is > 20 m from the GPS point — the classifier checks `snap_far` before `nominatim_overrode`, and the snap distance after `adoptCandidateAsSnap` represents how far the resolver reached to find Nominatim's street, not a snap-quality signal.

## Top failure signatures (code-structural)

1. `snap_far` masking `nominatim_overrode` — ordering bug at `lib/parking-quality-diagnose.ts:131–132`
   - Shared pattern: Nominatim override path (`adoptCandidateAsSnap`, `findCenterlineSegmentByName`) stores the adopted street's centerline distance in `snapResult.snapDistanceMeters` (→ `snap_distance_meters` DB column). That distance can be 25–150 m. The classifier hits the `snap_far` guard first (line 131) and returns early, never reaching the `nominatim_overrode` check (line 132).
   - In-code citation: `check-parking.ts:2159` (comment): *"the reported snap_distance is essentially how far we had to reach to find Nominatim's name — not how confident the geometry is."*

2. `snap_far` mislabeling when Nominatim _confirmed_ the snap — missing suppressor at `lib/parking-quality-diagnose.ts:131`
   - Shared pattern: When Nominatim agrees with the snap (orientation match → `nominatim_overrode = false`, `nominatim_agreed = true`), the original snap distance is kept in `snapResult`. If that distance is > 20 m (off-grid parking — back lot, wide ROW, setback), the classifier fires `snap_far`. But the code comment at line 123–127 explicitly states: *"Only flag snap_far when no cross-source confirmation exists … When Mapbox-reverse independently agrees with snap (and Nominatim too), a 25–40 m snap distance just means the user parked off-grid — the answer is correct, not a failure."* Nominatim confirmation (`nominatim_overrode === false`) is a valid cross-source signal but is not used as a suppressor.

## Root-cause hypotheses

### Hypothesis 1 — snap_far ordering (lines 131–132)

```typescript
// lib/parking-quality-diagnose.ts — current
const mb = row.native_meta?.mapbox_reverse;
const mapboxConfirmsSnap = mb && mb.matched && mb.agrees_with_snap === true;
if ((row.snap_distance_meters || 0) > 20 && !mapboxConfirmsSnap) return 'snap_far'; // ← fires first
if (row.nominatim_overrode) return 'nominatim_overrode';                             // ← never reached
```

When Nominatim overrides the snap and `adoptCandidateAsSnap` is called (`check-parking.ts:1891, 1937, 1992, 2029`), the function returns a new `snapResult` whose `snapDistanceMeters` is the adopted street's PostGIS centerline distance. This can be 25–150 m (`findCenterlineSegmentByName` searches up to 200 m radius). The DB column `snap_distance_meters` is set from `snapResult.snapDistanceMeters` at `check-parking.ts:3228`. The classifier then sees `snap_distance_meters > 20` and hits the `snap_far` branch before it can check `nominatim_overrode`.

Result: every `nominatim_overrode` event where Nominatim's adopted street was > 20 m away is silently misclassified as `snap_far`. The `nominatim_overrode` bucket is under-counted; `snap_far` is over-counted.

### Hypothesis 2 — missing Nominatim-confirmation suppressor (line 131)

When the Nominatim orientation matches the snap orientation (`check-parking.ts:2064–2069`), the code writes `nominatim_overrode = false` and `nominatim_agreed = true`. The snap is kept as-is. If the original snap was > 20 m (e.g., parking in a set-back lot, alley-adjacent, or wide arterial ROW), the classifier returns `snap_far` — but both snap and Nominatim agree, so this is not a pipeline failure. The code comment at line 123–127 of `parking-quality-diagnose.ts` explicitly names this as a false-positive case for the Mapbox path; Nominatim agreement provides the same suppression signal.

## Proposed patches

### Patch 1 — snap_far / nominatim_overrode ordering + Nominatim suppressor

- **Signatures addressed:** `snap_far` (over-counted), `nominatim_overrode` (under-counted)
- **Expected impact:** Events where Nominatim correctly overrode a bad snap move from `snap_far` → `nominatim_overrode`, giving operators visibility into how often the override fires. Events where Nominatim confirmed a far-but-correct snap move from `snap_far` → `healthy` or a lower-priority bucket. Total event count is unchanged — rows shift between buckets.
- **Evidence base:** In-code comment at `check-parking.ts:2159` explicitly documents that the stored `snap_distance_meters` after a Nominatim override is not a confidence signal. The code comment at `parking-quality-diagnose.ts:123–127` explicitly names Nominatim + Mapbox combined confirmation as the correct suppressor; Nominatim alone (`nominatim_overrode === false`) is the analogous signal for the Nominatim-only path.
- **File + line:** `lib/parking-quality-diagnose.ts:129–132`
- **Change shape:** Move the `nominatim_overrode` guard before the `snap_far` guard (1 line up). Add `const nominatimConfirmsSnap = row.nominatim_overrode === false;` (strict equality — `null`/`undefined` means Nominatim was never consulted, only explicit `false` means confirmed). Add `&& !nominatimConfirmsSnap` to the `snap_far` guard.

```diff
-  const mb = row.native_meta?.mapbox_reverse;
-  const mapboxConfirmsSnap = mb && mb.matched && mb.agrees_with_snap === true;
-  if ((row.snap_distance_meters || 0) > 20 && !mapboxConfirmsSnap) return 'snap_far';
-  if (row.nominatim_overrode) return 'nominatim_overrode';
+  if (row.nominatim_overrode) return 'nominatim_overrode';
+  const mb = row.native_meta?.mapbox_reverse;
+  const mapboxConfirmsSnap = mb && mb.matched && mb.agrees_with_snap === true;
+  const nominatimConfirmsSnap = row.nominatim_overrode === false;
+  if ((row.snap_distance_meters || 0) > 20 && !mapboxConfirmsSnap && !nominatimConfirmsSnap) return 'snap_far';
```

- **Test plan:** After merging, run `npx tsx scripts/parking-quality-diagnose.ts dotenv_config_path=.env.local hours=168`. Expect:
  - `nominatim_overrode` count increases (previously masked as `snap_far`)
  - `snap_far` count decreases by the same total
  - `healthy` count may increase slightly (far-but-Nominatim-confirmed snaps escape `snap_far`)
  - No change to any detection output — this is observability only
- **Risk:** Very low. Detection logic (`check-parking.ts`) is untouched. Only the post-hoc diagnostic classifier changes. The only way this goes wrong is if `nominatim_overrode` is accidentally `false` for rows where Nominatim was never consulted — but the code only writes `nominatim_overrode = false` at explicit confirmation points (`check-parking.ts:1980, 2069, 2231`), never as a default.
