## Headline finding

The `classify()` function in `lib/parking-quality-diagnose.ts` evaluates `snap_far` **before** `nominatim_overrode`. Since commit 2e6743b added the `two_source_intersection_override` path, all successful Nominatim overrides at intersections produce rows where `snap_distance_meters > 20m` AND `nominatim_overrode = true`. Under the current precedence, those rows are classified as `snap_far` instead of `nominatim_overrode`, masking the new code path from quality analysis and inflating the `snap_far` bucket.

## Root cause ‚Äî classifier precedence

**File:** `lib/parking-quality-diagnose.ts:115‚Äì141`

```typescript
function classify(row: any): FailureSignature {
  ...
  if (!row.snap_street_name) return 'no_snap';
  const mapboxConfirmsSnap = mb && mb.matched && mb.agrees_with_snap === true;
  if ((row.snap_distance_meters || 0) > 20 && !mapboxConfirmsSnap) return 'snap_far';  // ‚Üê line 131
  if (row.nominatim_overrode) return 'nominatim_overrode';                              // ‚Üê line 132
```

**Why this is wrong after commit 2e6743b:**

The `twoSourceOverride` branch (added 2026-04-30) fires when:
- `near_intersection = true`
- Apple Maps thoroughfare agrees with Nominatim's street name
- Apple Maps disagrees with the snap winner

When this branch fires, the code calls `adoptCandidateAsSnap(wideCandidate, 'two_source_intersection_override')` and replaces `snapResult` with the adopted candidate. That candidate is the **perpendicular street at the corner** ‚Äî whose centerline is typically 25‚Äì60m from the GPS point (the user parked near the corner, far from the perpendicular street's centerline). The DB row therefore records:
- `snap_distance_meters = 25‚Äì60` (the adopted perpendicular street, not the original wrong snap)
- `nominatim_overrode = true`

With the current precedence, `snap_distance_meters > 20 && !mapboxConfirmsSnap` fires first and returns `snap_far`. The classifier never reaches the `nominatim_overrode` check.

**When `nominatim_overrode = true`, the large snap distance is expected behavior, not a failure.** The snap was overridden precisely because the closest centerline was the wrong street. Classifying these rows as `snap_far` is:
1. Misleading ‚Äî the address is likely correct (three signals agreed: Nominatim + Apple + PostGIS candidate)
2. Inflating `snap_far` counts with events that are successful disambiguations
3. Hiding `nominatim_overrode` counts from the quality report, making it impossible to measure how often the new intersection-override path fires and whether it is producing good addresses

**Other Nominatim override paths affected:** The same masking applies to any `nominatim_overrode` row where the adopted candidate is >20m from GPS (e.g., `nominatim_override_extended`, `nominatim_override` when `findCenterlineSegmentByName` succeeds with a segment 20‚Äì150m away).

## Proposed patch

### Patch 1 ‚Äî move `nominatim_overrode` check before `snap_far` in classifier

- **Signature addressed:** `snap_far` (inflated) / `nominatim_overrode` (deflated)
- **File + line:** `lib/parking-quality-diagnose.ts:131‚Äì132`
- **Evidence base:** Static code analysis of `classify()` vs. `adoptCandidateAsSnap()` in commit 2e6743b. No live row IDs available this run.
- **Change shape:** Swap lines 131 and 132 so `nominatim_overrode` is checked before `snap_far`. When `nominatim_overrode = true`, a large snap distance means the override adopted a far-but-correct candidate; classifying as `nominatim_overrode` is accurate. The `snap_far` path still fires for all non-overridden rows where the snap was genuinely far without cross-source confirmation.
- **Expected impact:** `nominatim_overrode` count rises (previously hidden as `snap_far`); `snap_far` count falls by the same amount. Ratio change gives us the first real visibility into how often `two_source_intersection_override` is firing since 2026-04-30.
- **Test plan:** After deploying, re-run the diagnose script for the prior 7 days. `nominatim_overrode` should jump. The ratio `nominatim_overrode / (nominatim_overrode + snap_far)` tells us what fraction of `snap_far` counts were actually successful intersections override events.
- **Risk:** Low. This is a metrics reclassification, not a change to detection logic. No parking address is changed. The only risk is if we inadvertently preferred `nominatim_overrode` for a row that was genuinely a bad snap (both conditions true). In that case, the row is still more accurately described as an override event ‚Äî the snap was the original close/wrong snap, not the adopted candidate.
