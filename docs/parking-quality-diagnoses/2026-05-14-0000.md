## Headline regression

Static analysis identified a classifier bug in `lib/parking-quality-diagnose.ts`
that inflates the `compass_missing` failure count for rows that actually had a
valid trajectory-derived heading.

## Root-cause hypotheses

### Hypothesis 1 — trajectory rows classified as `compass_missing`

**Code path traced statically:**

`check-parking.ts` lines 772–779 handle the case where there is no per-fix
GPS heading (`hasHeading = false`) but drive-trajectory is available:

```
} else if (trajectoryMeanHeadingDeg != null) {
    effectiveHeading = trajectoryMeanHeadingDeg;
    effectiveHeadingSource = 'gps';
```

Because `hasHeading` stays false, the row is persisted with:
- `heading_source = 'gps'`  (from trajectory)
- `gps_heading = null`       (check-parking.ts line 797: `hasHeading ? headingDeg : null`)
- `effective_heading = <non-null trajectory mean>`

The classifier in `lib/parking-quality-diagnose.ts` line 149 then evaluates:

```
if (!row.compass_heading && !row.gps_heading) {
  // → true for trajectory rows
  const snapConfirmedByMapbox = …;
  if (!snapConfirmedByMapbox) return 'compass_missing';  // ← fires incorrectly
}
```

The classifier never checks `row.heading_source` or `row.effective_heading`
before firing `compass_missing`, so any trajectory row whose snap isn't
independently confirmed by Mapbox-reverse gets mislabeled.

**File:line:** `lib/parking-quality-diagnose.ts:149`

**Impact estimate:** Every iOS session that used trajectory median heading but
lacked a per-fix GPS heading would inflate `compass_missing`. Trajectory fallback
is the path taken for CarPlay-disconnected sessions and some stop-start captures
where the GPS fix itself had no heading field. Likely affects a non-trivial
fraction of iOS auto-detect events.

## Proposed patches

### Patch 1 — guard `compass_missing` on `heading_source`

- **Signature addressed:** `compass_missing` (classifier false positives)
- **Expected impact:** Reduces `compass_missing` count for trajectory-based rows.
  Rows that were healthy but mislabeled will now correctly reach `low_accuracy`
  or `healthy`. Does NOT change any detection logic — this is a diagnosis-only
  classifier file.
- **Evidence base:** Static code analysis only. No example row IDs available
  this run. Needs live data verification.
- **File + line:** `lib/parking-quality-diagnose.ts:149`
- **Change shape:** Replace the current guard

  ```ts
  if (!row.compass_heading && !row.gps_heading) {
  ```

  with one that also allows through only when `heading_source` is absent or
  explicitly `'none'` (meaning no heading of ANY kind was available):

  ```ts
  if (!row.compass_heading && !row.gps_heading
      && (!row.heading_source || row.heading_source === 'none')) {
  ```

  This is logically correct: `heading_source='gps'` means the pipeline used a
  GPS-family heading (per-fix or trajectory); classifying that row as
  `compass_missing` is a false positive by definition.

- **Test plan:** After deploying, re-run the diagnosis script over the last 7 days
  (`hours=168`). `compass_missing` count should drop. `healthy` count should
  rise by the same amount (or shift to `low_accuracy` if those rows also had
  low GPS accuracy). The delta is the former false-positive mass.
- **Risk:** If any older rows genuinely have `heading_source='gps'` but no
  actual heading was usable (e.g. a write-path bug that set heading_source
  incorrectly), those rows would escape the `compass_missing` label. Low risk
  because `heading_source` is written from `effectiveHeadingSource` which is
  only ever `'gps'` when `hasHeading` is true OR `trajectoryMeanHeadingDeg`
  is non-null — both indicate real heading data.
