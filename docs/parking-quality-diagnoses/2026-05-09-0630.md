## Top failure signature

### 1. `heading_stale` — permanently 0 (structural dead code)

**Evidence (code-based, not row-based):**

`lib/parking-quality-diagnose.ts:127`:
```typescript
if (row.heading_source === 'stale') return 'heading_stale';
```

`pages/api/mobile/check-parking.ts:743`:
```typescript
let effectiveHeadingSource: 'compass' | 'gps' | 'none';
```

The TypeScript union type proves `heading_source` is written as one of three literals —
`'compass'`, `'gps'`, or `'none'` — at lines 756, 761, 771, 775, 779. The string `'stale'`
does not appear as an assigned value anywhere in the server codebase (`grep` returns no
matches).

The stale-GPS guard fires at lines 747–758 when both GPS heading and compass are present,
they disagree by >30°, AND they classify to different grid orientations (the Belden/Kenmore
class of miss). When it fires, it sets `effectiveHeadingSource = 'compass'` and logs
"GPS heading likely stale after turn — preferring compass." The event is handled correctly.
But the metric classifier never sees it as `heading_stale` — it appears as `compass_missing`
or `healthy` depending on other conditions.

**Shared pattern across known examples:**
- Belden/Kenmore miss (2026-05-08, diag #173): GPS heading 5° (N-S, stale from driving
  Kenmore north), compass 98° (E-W, parked facing east on Belden). System correctly chose
  compass over GPS upstream. But `heading_source` was written as `'compass'`, not `'stale'`.
  The report sees this as a normal compass-disambiguated park, not a stale-GPS event.
- Commit `7fcf483e` (2026-05-08) fixed a downstream Nominatim override gate that failed to
  check compass. The upstream stale-heading label was already wrong; the fix addressed the
  symptom, not the invisible metric.

## Root-cause hypothesis — heading_stale

The stale-heading detection logic is correct and handles the Belden/Kenmore class of miss. The
problem is purely metric observability: the classifier in `lib/parking-quality-diagnose.ts` looks
for a string that is never produced. Because the bucket is always 0, there is no signal in the
quality report for operators to know how often the stale-GPS fallback is being used, or whether it
is trending up (which would suggest GPS quality is degrading or users are parking faster after
turns).

The fix is:
1. Persist a boolean `gps_heading_was_stale: true` in `native_meta` when the stale guard fires.
2. Update the classifier to read this field instead of the impossible `heading_source === 'stale'`.

No schema migration needed — `native_meta` is an existing JSONB column.
No detection behavior changes — only metric observability improves.
