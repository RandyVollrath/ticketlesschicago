### Observation 1 — `snap_far` classifier threshold vs. wide-ROW fallback

**Files:** `lib/parking-quality-diagnose.ts:131`, `pages/api/mobile/check-parking.ts:1172-1186`

The classifier marks a row as `snap_far` when `snap_distance_meters > 20` AND
Mapbox-reverse does not confirm the street. However, `check-parking.ts` has a
"wide-ROW fallback" that deliberately accepts snaps up to 60m when the closest
candidate has a ≥15m geometric gap to the next competitor (the `geom_gap` path).
When this fallback fires on a wide street (e.g., Southport at 2472N, cited in a
comment at line 1164), the snap is geometrically correct — but the classifier
still labels it `snap_far` because only the `apple_confirmed` path sets
`mb.agrees_with_snap = true` in the Mapbox metadata.

**Potential effect:** wide-ROW `geom_gap` snaps inflate `snap_far` counts in the
quality report even when the answer is correct. If `snap_far` is a top signature,
some fraction of it may be false positives from this fallback path.

**What would confirm this:** look for `snap_far` example rows where
`diag.snap_wide_row_reason === 'geom_gap'` in `native_meta`. If the
`resolved_address` matches the user's actual street, it's a false-positive signal.

**Proposed change shape (if confirmed):** in `classify()`, add a check for
`row.native_meta?.snap_wide_row_fallback === true` — when the wide-ROW fallback
fired and was the `geom_gap` variant, treat the row the same as Mapbox confirmation
(i.e., don't flag `snap_far`). One-line change to the `snap_far` condition in
`lib/parking-quality-diagnose.ts:131`.

### Observation 2 — `heading_stale` vs. walkaway-guard threshold gap

**Files:** `lib/parking-quality-diagnose.ts:141-144`, `pages/api/mobile/check-parking.ts:2931`

The classifier marks `heading_stale` when `heading_source === 'gps'` and
`headingDisagreementDeg > 30`. The walkaway guard that *suppresses notifications*
uses a higher threshold of `> 50°`. This means there is a 30–50° disagreement
band where:
- The classifier labels the row `heading_stale` (potentially a quality failure), AND
- Notifications were still sent (walkaway guard did not fire).

Whether the 30° classifier threshold is right depends on whether GPS-heading rows
with 30–50° compass disagreement actually produce wrong addresses. If they mostly
produce correct addresses (GPS heading was stale but snap geometry was close
enough to still pick the right street), then `heading_stale` is over-counting
failures in this band.

**What would confirm this:** look for `heading_stale` rows with
`headingDisagreementDeg` between 30 and 50 and check `street_correct` values. If
street_correct is predominantly true in that band, the `> 30` threshold in the
classifier is too aggressive.

**Proposed change shape (if confirmed):** raise the classifier threshold from 30°
to 45° (still below the 50° walkaway cutoff), matching the band where heading
disagreement actually causes wrong-street outcomes. Change in
`lib/parking-quality-diagnose.ts:143`.

### Observation 3 — `maxSnapDistance` dynamic floor vs. low-accuracy GPS

**Files:** `pages/api/mobile/check-parking.ts:1158`

```typescript
const maxSnapDistance = accuracyMeters ? Math.max(accuracyMeters, 30) : 40;
```

When the device reports GPS accuracy of, say, 45m, `maxSnapDistance` becomes 45m.
A candidate at 42m is included. The classifier then flags it `snap_far` (> 20m,
no Mapbox confirmation). This is correct behavior — we accepted a wide snap
because GPS uncertainty justifies it — but the `snap_far` label may mask a
`low_accuracy` root cause. In the quality report these would appear as `snap_far`
events, not `low_accuracy` events, potentially underreporting the low-accuracy
signature.

**What would confirm this:** `snap_far` rows where `raw_accuracy_m > 30` AND
`snap_distance_m` is between 20m and `raw_accuracy_m`. These are double-counted —
they appear in `snap_far` but are really driven by GPS quality.

**Proposed change shape (if confirmed):** in `classify()`, check if
`snap_distance_m <= raw_accuracy_m` when classifying `snap_far`. If so, classify
as `low_accuracy` instead — the snap distance is within the GPS uncertainty
envelope, so the snap is not actually "far" relative to what the hardware can
provide. Change in `lib/parking-quality-diagnose.ts:131`.
