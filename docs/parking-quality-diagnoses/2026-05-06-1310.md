## Static code analysis findings

### Finding 1 — `snap_far` classifier may over-fire when Mapbox token is absent

**File:** `lib/parking-quality-diagnose.ts:131`

```typescript
const mb = row.native_meta?.mapbox_reverse;
const mapboxConfirmsSnap = mb && mb.matched && mb.agrees_with_snap === true;
if ((row.snap_distance_meters || 0) > 20 && !mapboxConfirmsSnap) return 'snap_far';
```

When `MAPBOX_ACCESS_TOKEN` is not set (or rate-limited), `lib/mapbox-reverse-geocode.ts:44`
returns `empty('no_token')` immediately. The diagnostic row's `native_meta.mapbox_reverse`
will have `matched: false`, so `mapboxConfirmsSnap = false` for every row.

In that state, **every snap farther than 20 m is labeled `snap_far`**, even when
Nominatim independently agreed with the snap street name. The classifier does not
consult Nominatim agreement (`diag.nominatim_agreed`) as a fallback confirmation
source.

**Potential impact:** Inflated `snap_far` counts in periods where Mapbox is
unavailable. The `nominatim_overrode` check runs *after* `snap_far` in the
classifier's precedence order (line 132 vs 131), so a Nominatim-confirmed snap
at 25 m still gets `snap_far` rather than something less alarming.

**Proposed fix shape** *(actionable only after live data confirms Mapbox is
actually being skipped in production)*:

Extend the `mapboxConfirmsSnap` guard to also check Nominatim agreement:

```typescript
// lib/parking-quality-diagnose.ts:129-131
const mb = row.native_meta?.mapbox_reverse;
const mapboxConfirmsSnap = mb && mb.matched && mb.agrees_with_snap === true;
const nominatimConfirmsSnap =
  row.nominatim_overrode === false && !!row.nominatim_street && !!row.snap_street &&
  row.nominatim_street === row.snap_street;
if ((row.snap_distance_meters || 0) > 20 && !mapboxConfirmsSnap && !nominatimConfirmsSnap)
  return 'snap_far';
```

**Risk:** Slightly under-counts `snap_far` in real geometric failures where Nominatim
also agrees with a wrong snap. Low risk because Nominatim agreement is an independent
signal from a different dataset.

**Evidence needed before implementing:** Pull `overall_failure_counts.snap_far` from
a fresh run, then count how many of those rows have `native_meta.mapbox_reverse.skip_reason = 'no_token'`
AND `nominatim_street = snap_street`. If that count is > 0, this is actionable.

### Finding 2 — `mobile_ground_truth_events` migration may not be applied in production

**Commit:** `cdcf7ee` (2026-05-03) — *"Mobile ground-truth events table + truth_metrics surfacing"*

The commit message explicitly warns:
> ⚠️ MIGRATION REQUIRED: `supabase/migrations/20260430_create_mobile_ground_truth_events.sql`
> must be applied to prod before this code paths fire. Verified missing as of this commit.
>
> Per-table comment in the migration: `/api/mobile/ground-truth` has been inserting
> into a non-existent table since the correction UI shipped — the endpoint catches
> the error and returns `success:true`, so every user correction has been silently
> dropped and `lookupUserParkingAnchor` (the tie-breaker in check-parking) has been
> returning null forever.

The `ground-truth.ts` endpoint (line 56-58) still catches insert failures silently
and returns `success: true, accepted: 0` with a `warning: 'insert_failed'` flag. If
the migration was never run, **all user parking corrections since the correction UI
shipped have been silently discarded.**

Additionally, there are two migration files for this table:
- `supabase/migrations/20260216074000_create_mobile_ground_truth_events.sql` (Feb 2026)
- `supabase/migrations/20260430_create_mobile_ground_truth_events.sql` (Apr 2026)

Both use `CREATE TABLE IF NOT EXISTS`, so double-applying is safe, but the existence
of both suggests the Feb migration may never have been applied. The April migration
adds the spatial composite index `idx_mobile_ground_truth_events_user_latlng` that
the Feb one lacks — this index is required for fast `lookupUserParkingAnchor` queries.

**Impact on parking quality:** If the table does not exist (or exists without data
because corrections were dropped before the table was created), `lookupUserParkingAnchor`
always returns null. At near-intersection parkings the anchor tie-breaker can't help
the cascade pick the right street, meaning we rely purely on geometry + heading. This
raises the expected error rate at corners by a small but nonzero margin.

**What to verify manually:**
```sql
-- In Supabase SQL editor:
SELECT count(*) FROM mobile_ground_truth_events;
-- If this throws "relation does not exist", the migration was never applied.
-- If count = 0 and the correction UI has been live for months, corrections were dropped.
```

**This is not a code bug** — no code change is needed. This is an ops check.
