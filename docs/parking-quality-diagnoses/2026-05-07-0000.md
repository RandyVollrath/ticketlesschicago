## Code inspection — top findings

### Finding 1 — `heading_stale` classifier never fires

**File:** `lib/parking-quality-diagnose.ts:127`
```typescript
if (row.heading_source === 'stale') return 'heading_stale';
```

**Why it's dead:** `heading_source` is written by the server at `pages/api/mobile/check-parking.ts:3142`:
```typescript
heading_source: effectiveHeadingSource,
```

`effectiveHeadingSource` is typed as `'compass' | 'gps' | 'none'` and is only ever set to one
of those three values (lines 756, 761, 771, 775, 778). The stale-heading guard fires at lines
747–758: when `headingDisagreementDeg > 30°` AND GPS and compass classify to different grid
orientations, the server silently falls back to compass (`effectiveHeadingSource = 'compass'`)
and writes that to the diagnostic. The string `'stale'` is never written anywhere in server
code (confirmed by grep — zero matches).

**Impact:** Every detection where GPS heading was detected as stale-after-a-turn is silently
reclassified as `compass_missing` (if no compass), or falls through to `low_accuracy` / `healthy`.
The `heading_stale` bucket has always been 0 and the metric is meaningless.

**Evidence basis:** Code-read only (no live rows). The dead-code analysis is conclusive from
reading the source — no live data needed to confirm the mismatch.

## Root-cause hypotheses

### Hypothesis for Finding 1

The `heading_stale` signature was added to the classifier expecting the server to write
`heading_source = 'stale'` when the stale guard triggered. Instead, the server writes the
**resolved** effective source (compass fallback). The original design intent was probably to
record the fact that GPS was discarded, but the diagnostic field was never added — only the
fallback logic was.

This is a measurement gap, not a detection bug: the stale-heading logic itself is correct
(it discards GPS and falls back to compass). The problem is we have no visibility into how
often it fires.

## Proposed patches

### Patch 1 — Surface `gps_heading_was_stale` in native_meta

- **Signature addressed:** `heading_stale` (currently dead)
- **Expected impact:** The `heading_stale` bucket starts reporting real counts; no user-facing
  behaviour changes
- **Evidence base:** Code-read only. Needs live data validation (confirm 0 rows currently have
  `heading_source='stale'`) before implementing
- **File + line:** `pages/api/mobile/check-parking.ts` at the stale guard branch (lines 753–758),
  and `lib/parking-quality-diagnose.ts:127`
- **Change shape:**
  1. In `check-parking.ts` lines 753–758: add `diag.native_meta = { ...(diag.native_meta || {}), gps_heading_was_stale: true };` inside the stale-guard `if` block (no schema change — stored in the existing `native_meta` JSONB column)
  2. In `lib/parking-quality-diagnose.ts:127`: change the classifier to `if (row.native_meta?.gps_heading_was_stale === true) return 'heading_stale';`
- **Test plan:** Deploy + run diagnosis for 24h. `heading_stale` count should become non-zero
  (or near-zero, confirming the guard rarely fires). Compare vs `compass_missing` count which
  should drop by the same delta.
- **Risk:** Stale-heading events that were previously silently handled by compass fallback will
  now appear as `heading_stale` failures — the dashboard failure rate will appear to rise
  slightly. This is accuracy improvement (we're measuring something real), not a regression.
