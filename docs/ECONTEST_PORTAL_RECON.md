# eContest Portal Recon (2026-04-23)

## Portal URL
`https://parkingtickets.chicago.gov/EHearingWeb/home`

Linked from: Chicago Admin Hearings → "Contesting Tickets" → "Get Started Online"

## Key Findings

### No Captcha
Zero captcha of any kind — no hCaptcha, no reCAPTCHA, no Turnstile. Form is completely unprotected.

### Technology Stack
- Java/Spring MVC (traditional server-rendered, NOT a SPA)
- CSRF token via hidden `_csrf` field (standard Spring Security)
- Session-based (cookies required)

### Flow (confirmed via Playwright)

**Step 1: Enter ticket numbers**
- URL: `/EHearingWeb/home`
- POST to: `/EHearingWeb/displayEligibleTickets`
- Fields: `ticket1`, `ticket2`, `ticket3` (up to 3 per submission)
- CSRF token from hidden field

**Step 2: Ticket eligibility display**
- URL: `/EHearingWeb/eligibleTicketsDisplay`
- Shows ticket details: number, violation, plate, state, date, amount, status
- Checkbox: "Check to Contest"
- Checkbox: "Add Evidence"
- Dropdown for contest method (seen: "Correspondence", "In-Person", "Virtual")
- Hidden fields carry all ticket data forward:
  - `confirmed[0].ticketNumber`
  - `confirmed[0].violDesc`
  - `confirmed[0].plateNumber`
  - `confirmed[0].plateState`
  - `confirmed[0].issueDateStr`
  - `confirmed[0].ticketAmountDue`
  - `confirmed[0].status`
  - `confirmed[0].statusAsString`
  - `confirmed[0].hearingDate`
  - `confirmed[0].noticeNumber`
  - `confirmed[0].contestMethod`
  - `confirmed[0].contestStatus` (boolean)
- "Continue" button submits

**Step 3: Evidence upload + confirmation (NOT YET PROBED)**
- Need a ticket in contestable state to reach this page
- Page text says: "If requesting a correspondence or virtual hearing, you will be required to submit any evidence you wish to present in defense of the violation."
- Likely has: file upload fields, text area for defense narrative, submit button

### Ticket Statuses Observed
- `liableDetermination` — already decided, not eligible for re-contest
- `determination` — already decided (camera tickets)
- Tickets with `portal_payable: true` + `hearing_end_date` in range may still be eligible
- Fresh tickets in "Notice" status should be eligible (we didn't have any in DB to test)

### Contest Methods Available
- **Correspondence** — mail-in equivalent, submitted online with evidence upload
- **In-Person** — schedule in-person hearing
- **Virtual** — WebEx virtual hearing (requires registered owner data from SOS)

## Architecture for Implementation

### Integration Points
- Ticket numbers already scraped by `lib/chicago-portal-scraper.ts` from payment portal
- Contest letters already generated by `scripts/autopilot-check-portal.ts`
- Letters already rendered as PDFs for Lob mailing
- Need: new submission path that uploads PDF + evidence to eContest instead of Lob

### Proposed Flow
1. After letter generation, attempt eContest submission first
2. If eContest fails (portal down, ticket ineligible, etc.), fall back to Lob mail
3. Store submission confirmation in `contest_letters` table (new columns)
4. Send user confirmation receipt

### Cost Savings
- Lob: ~$1.50/letter (postage + printing)
- eContest: $0
- At scale (100 letters/month): saves ~$150/month

### Risk Factors
- Portal could add captcha later (monitor for changes)
- Portal could change form structure (add format drift detection like payment scraper)
- Session management more complex than stateless API call
- "Correspondence" contest method win rate unknown (could be same as mail-in 59%, or different)
